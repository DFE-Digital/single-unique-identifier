
# DPIA – Data Store Catalogue and Data Inventory

## 1. Data Store Catalogue

| Data Store | Exists in Discovery | Description | Purpose |
|-----------|--------------------|-------------|---------|
| ID Register (SuiCustodianRegister) | Yes | Stores relationships between a person (NHS number) and custodians who have held records for them, including timestamps and provenance | Enables discovery of which organisations have held records for a person and supports operational lifecycle management |
| Jobs (Lease / Work Broker Table) | Yes | Stores units of work issued to custodians, including lease state, payload, and lifecycle timestamps | Enables distributed processing by allowing custodians to claim and process work safely using a lease model |
| Job Results Queue (Inbound) | Yes | Temporary queue receiving results submitted by custodians after processing a job | Decouples inbound requests from processing and ensures reliable ingestion |
| Search Results Table | Yes | Stores record pointers returned by custodians | Aggregates results across custodians into a single search response |
| Work Item Job Count | Yes | Stores expected job counts for a search | Enables calculation of search completeness |
| Custodian Directory | No (Proposed) | Stores organisation and contact details | Supports onboarding and governance |
| DSA Configuration / Policy Registry | No (Proposed) | Stores data sharing policies as an event-sourced stream of include and exclude rules, with rule-level lifecycle | Enables enforcement of data sharing decisions at FIND and FETCH boundaries using rule-based evaluation with a default deny posture |
| Webhook Registrations | No (Future Option) | Stores webhook subscription details | Enables push-based optimisation |
| Audit / Operational Logs (Application Insights) | Yes | Stores operational logs and audit-relevant events generated by services, including endpoint interactions, authentication activity, job processing, and policy enforcement outcomes | Supports monitoring, troubleshooting, security auditing, and verification of policy enforcement decisions |

##### Enforcement Statement

DSA policies are defined as collections of include and exclude rules. Rules are evaluated at request time with a default deny posture. Exclude rules take precedence over include rules. Rules are applied at the FIND and FETCH boundaries and are constrained by record type, destination, context, disclosure level, and validity period.

##### Enforcement Statement

Webhook registrations define which events a custodian may receive, where they are sent, and how delivery is authenticated. Delivery must only occur where the registration is active, within any validity period, and configured with a supported authentication method. Secrets such as API keys should be treated as security-sensitive configuration and stored using appropriate secret-management controls.

##### Logging and Retention Statement

Operational logs are designed to avoid storing personally identifiable information. Where identifiers are required for correlation, they are obfuscated or pseudonymised. Logs are retained in line with defined operational and security retention policies, balancing the need for auditability and investigation with data minimisation principles. Access to logs is restricted and monitored.
